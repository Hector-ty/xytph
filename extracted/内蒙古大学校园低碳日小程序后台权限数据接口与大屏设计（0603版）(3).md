[P:Heading 1] 一、设计边界与核心闭环

[P:Normal] 小程序必须服务活动方案中的六大专区和公共展示需求，所有页面、后台和数据接口都围绕低碳日当天可执行、可核验、可统计展开。

[P:List Bullet] 活动入口：学生通过公众号、学院群、食堂桌贴、宿舍海报、现场二维码进入小程序。

[P:List Bullet] 身份确认：学生完成微信登录并绑定学院、学号、姓名等校园身份；教师或嘉宾可作为普通参与者进入。

[P:List Bullet] 任务参与：学生在绿行、减塑、光盘、循环集市、低碳宿舍等专区选择任务，按规则扫码、上传照片、同步步数或提交承诺。

[P:List Bullet] 审核与核销：系统自动判定低风险任务，工作人员核销现场任务，后台审核照片和异常数据。

[P:List Bullet] 积分发放：所有积分通过积分流水入账，不允许前端直接改分；积分用于个人榜、学院榜、专区兑换和电子证书。

[P:List Bullet] 成果展示：现场大屏和后台数据导出展示参与人数、任务完成量、积分发放量、学院排行、答题数据和低碳贡献估算。

[P:Heading 1] 三、后台页面设计

[P:Heading 2] 3.1 活动配置页

[P:Normal] 页面入口：后台首页 -> 活动管理 -> 内蒙古大学校园低碳日。

## [TABLE 3]
| +------------------------------------------------------------+ | 活动配置 | | 活动名称 [内蒙古大学校园低碳日碳普惠主题活动] | | 活动主题 [绿色转型，全民同行] | | 活动时间 [开始] [结束] | | 预热时间 [开始] [结束] | | 活动状态 [草稿/预热/进行中/已结束 ▼] | | 是否开放签到 [开关] 是否开放兑换 [开关] 是否开放大屏 [开关] | | [保存] [发布] [结束活动] | +------------------------------------------------------------+ |

[P:Normal] 前端逻辑：后台前端展示活动基础信息和状态按钮。发布前校验任务、积分规则、奖品和二维码是否配置完整。

[P:Normal] 后端逻辑：后端保存 activity 表；活动状态控制学生端是否能签到、做任务、兑换和查看证书。发布动作写入操作日志。

[P:Normal] 采集数据：活动名称、主题、时间、地点、状态、开关、创建人、发布时间。

[P:Normal] 状态与提示：草稿不可被学生访问；进行中可做任务；已结束只读，允许导出和证书生成。

[P:Heading 2] 3.2 任务与积分规则配置页

[P:Normal] 页面入口：后台 -> 任务管理。

## [TABLE 4]
| +------------------------------------------------------------+ | 任务管理 | | [新增任务] [批量导入] [导出规则] | | 专区: 绿行/减塑/光盘/循环/宿舍/公共 | |------------------------------------------------------------| | 任务名 | 完成方式 | 积分 | 每日上限 | 审核方式 | 状态 | 操作 | | 步数挑战 | 自动 | 3/5/8 | 8分 | 自动 | 启用 | 编辑 | | 光盘午餐 | 上传 | 10 | 1次 | 人工 | 启用 | 编辑 | +------------------------------------------------------------+ |

[P:Normal] 前端逻辑：前端支持配置任务名称、专区、完成方式、积分规则、上限、是否需要审核、是否进入排行榜、是否接入大屏。

[P:Normal] 后端逻辑：后端写入 tasks、task_rules。积分规则要结构化存储，供积分引擎读取，不建议把规则写死在前端。

[P:Normal] 采集数据：task_id、zone、complete_type、points_rule、daily_limit、audit_required、rank_enabled、screen_enabled。

[P:Normal] 状态与提示：规则修改后只影响新提交任务；已发放积分不自动重算，除非管理员执行回滚/补发。

[P:Heading 2] 3.3 二维码点位管理页

[P:Normal] 页面入口：后台 -> 点位二维码。

## [TABLE 5]
| +------------------------------------------------------------+ | 二维码点位管理 | | [新增点位二维码] | | 点位名称 [食堂午餐光盘点] | | 绑定任务 [光盘午餐 ▼] | | 有效时间 [11:00-13:30] | | 二维码类型 [签到/任务/起点/终点/兑换/展板 ▼] | | [生成二维码] [下载海报] | +------------------------------------------------------------+ |

[P:Normal] 前端逻辑：后台生成二维码后可下载二维码图片或二维码海报。二维码列表显示状态、扫描次数、最后扫描时间。

[P:Normal] 后端逻辑：后端生成带签名参数的二维码，字段包含 qr_id、activity_id、task_id、point_type、expires_at、signature。可停用二维码。

[P:Normal] 采集数据：qr_id、二维码类型、绑定任务、点位、有效期、签名、状态、扫描次数。

[P:Normal] 状态与提示：过期或停用二维码返回不可用；活动当天可一键延长有效期；二维码泄露时可停用并重新生成。

[P:Heading 2] 3.4 审核中心

[P:Normal] 页面入口：后台 -> 审核中心。

## [TABLE 6]
| +------------------------------------------------------------+ | 审核中心 | | 筛选: 专区[光盘] 状态[待审核] 学院[全部] | |------------------------------------------------------------| | 学生 | 学院 | 任务 | 提交时间 | 图片 | 操作 | | 张三 | 法学院 | 午餐光盘 | 12:20 | [预览] | [通过] [驳回] | | 驳回原因: [照片模糊/重复图片/与任务无关/其他] | +------------------------------------------------------------+ |

[P:Normal] 前端逻辑：审核员可以按专区、任务、学院、状态筛选。图片预览要支持放大、查看原图和上一条/下一条快速审核。

[P:Normal] 后端逻辑：审核通过时调用积分引擎发分；驳回时记录原因并通知学生。所有审核动作写入 audit_logs。

[P:Normal] 采集数据：submission_id、审核状态、审核人、审核时间、驳回原因、积分流水ID。

[P:Normal] 状态与提示：已通过记录不可普通撤销，需要管理员执行积分回滚；重复图片可通过 hash 或人工判断标记异常。

[P:Normal] 页面入口：后台 -> 奖品管理/兑换记录。

## [TABLE 7]
| +------------------------------------------------------------+ | 奖品管理 | | 奖品名 | 所需积分 | 积分类型 | 库存 | 状态 | 操作 | | 食堂券 | 20 | 光盘/减塑 | 100 | 上架 | 编辑/下架 | | 骑行券 | 8 | 绿行 | 50 | 上架 | 编辑/下架 | |------------------------------------------------------------| | 兑换记录 | | 学生 | 奖品 | 兑换码 | 状态 | 核销人 | 核销时间 | +------------------------------------------------------------+ |

[P:Normal] 前端逻辑：奖品管理员维护库存、兑换上限、适用积分类型、奖品说明和兑换时间窗口。兑换记录支持按状态筛选。

[P:Normal] 后端逻辑：兑换申请和核销必须使用数据库事务。库存扣减、积分扣减和兑换记录状态变更要么全部成功，要么全部失败。

[P:Normal] 采集数据：reward_id、库存、兑换上限、积分类型、兑换码、核销状态、核销人、核销时间。

[P:Normal] 状态与提示：库存不足不能生成兑换码；已生成未核销可设置过期时间；现场异常可管理员取消兑换并退回积分。
