# 内蒙古大学校园低碳日小程序完整开发规格与 AI 一次性交付提示词

版本：V2.0（完整交付版）  
日期：2026-06-05  
适用对象：产品经理、小程序前端、后台前端、后端、测试、运维、AI 编程助手  
系统范围：学生端微信小程序、工作人员端、管理后台、现场数据大屏、后端服务与数据接口

> 本文档是五份 0603 版材料的统一开发口径，只描述软件系统。活动仪式、物料准备、线下场地安排等内容不在本文档范围内；线下点位、工作人员和奖品仅作为系统配置或外部依赖出现。本文档要求完整交付全部软件范围，不分一期、二期或其他开发阶段，不接受仅实现部分功能的交付结果。

## 1. 文档执行规则

### 1.1 规格冲突处理

发生冲突时，按以下顺序执行：

1. 本文档“锁定业务规则”和验收标准。
2. 后台已发布的活动、任务、积分、奖品和二维码配置。
3. 页面展示文案与视觉稿。
4. 开发人员或 AI 的实现假设。

任何未明确事项不得由前端写死。确需暂定时，使用可配置默认值并在交付说明中列出。

### 1.2 技术栈原则

优先沿用现有代码仓库和团队技术栈，不因本文档重构已可用的基础设施。若项目尚未创建，可采用以下默认实现：

| 层级 | 默认实现 | 可替换要求 |
| --- | --- | --- |
| 学生端/工作人员端 | 微信原生小程序 + TypeScript | 可替换为团队已验证的小程序框架，但必须保留微信登录、扫码、上传和分包能力。 |
| 管理后台/大屏 | 响应式 Web 应用 | 后台与大屏可共用工程，但权限和发布入口必须隔离。 |
| 后端 | REST JSON API，单体模块化优先 | 不强制使用微服务；模块边界、事务和权限必须清晰。 |
| 数据库 | MySQL 8.x | 必须支持事务、唯一约束、索引和行级锁。 |
| 缓存 | Redis | 用于排行榜、大屏快照、限流和短期幂等，不作为唯一事实来源。 |
| 图片存储 | 对象存储 | 数据库只保存受控 URL、哈希、大小和元数据。 |
| 时区 | Asia/Shanghai | 服务端时间为准，接口时间使用 ISO 8601。 |

微信能力接入前，应按微信官方最新文档核验 `wx.login`、步数能力、扫码、小程序码和上传接口的权限、隐私与审核要求。若步数能力无法开通，绿行专区必须保留“凭证上传或现场扫码”的降级路径。

### 1.3 系统明确不做

- 不做社交动态、关注、评论、私聊或复杂社区功能。
- 不做人民币支付、积分转赠、积分交易或可交易碳资产。
- 不做复杂碳核算平台；减碳量仅可作为可配置的估算展示。
- 不做持续 GPS 轨迹采集、面部识别或非必要敏感信息采集。
- 不让学生自行录入循环集市物品积分。
- 不允许前端传入或决定最终积分值。

## 2. 产品目标与软件闭环

### 2.1 完整交付目标

在校园低碳日活动期间，系统稳定完成以下闭环：

1. 微信登录并绑定校园身份。
2. 电子签到并记录入口来源。
3. 学生完成任务并提交证据。
4. 系统自动判定、工作人员核销或审核员审核。
5. 后端积分引擎发放可审计积分。
6. 学生查看积分、排行榜并申请兑换。
7. 工作人员核销兑换码，库存与积分原子更新。
8. 后台、大屏和导出使用同一数据口径。

### 2.2 系统组成

| 子系统 | 主要用户 | 主要职责 |
| --- | --- | --- |
| 学生端小程序 | 学生、教师、嘉宾 | 登录认证、签到、做任务、上传凭证、查看积分、兑换、答题、查看排行榜和证书。 |
| 工作人员端 | 志愿者、摊位人员、奖品管理员 | 扫描学生码、循环登记、现场任务核验、兑换核销、查看本人操作记录。 |
| 管理后台 | 活动管理员、审核员、奖品管理员、数据查看者 | 配置活动和任务、生成二维码、审核、库存、异常处理、导出和日志。 |
| 数据大屏 | 老师、领导、展示人员 | 展示脱敏汇总指标、学院榜、任务热度和专区数据。 |
| 后端服务 | 所有客户端 | 权限、业务规则、状态机、积分、事务、审计、缓存和数据接口。 |

## 3. 锁定业务规则

### 3.1 积分口径

- 积分是校园活动激励，不等同人民币，不可转赠或交易。
- 每个任务只向一个积分账户类型发放积分：`GENERAL`、`GREEN_TRAVEL`、`FOOD_PLASTIC`、`RECYCLE`、`DORM` 或 `KNOWLEDGE`。
- `rank_total` 为用户活动期间所有有效正向任务积分之和，扣除撤销和回滚，但不因兑换奖品而减少。
- `available_balance` 为各账户当前可用余额，会因兑换、管理员扣减或回滚而变化。
- 排行榜使用 `rank_total` 或指定专区累计有效积分；兑换使用奖品配置的指定账户余额。
- 所有积分变化必须写入不可直接修改的积分流水；纠错使用反向流水，不更新历史流水金额。
- 同一业务来源只能产生一次同类型积分效果，使用唯一约束防止重复发分。

### 3.2 任务与审核

- 前端只提交行为证据和业务参数，最终积分由后端读取已发布规则计算。
- 自动任务可直接完成；照片、凭证和随手拍任务进入待审核；现场任务由有权限工作人员核销。
- 规则修改只影响修改后新建的任务记录；历史积分不自动重算。
- 驳回必须填写标准原因或补充说明，并按任务配置决定是否允许重新提交。
- 图片任务必须保存图片哈希、提交时间、提交人、任务、审核人、审核结果和驳回原因。
- 同一任务的次数限制、每日上限和专区积分上限均由后端校验。

### 3.3 二维码与核销

- 二维码必须绑定活动、类型、点位、有效时间和签名；任务码还必须绑定任务。
- 二维码类型包括：签到、任务、健步起点、健步终点、工作人员入口、学生个人码、兑换核销和展板题组。
- 过期、停用、签名错误、活动不匹配或点位不匹配的二维码不得继续业务流程。
- 健步走必须先扫起点再扫终点，并校验时间窗口。
- 兑换码一次性使用；核销失败必须返回明确原因。

### 3.4 身份与隐私

- 未认证用户可查看公开活动介绍和规则，但不能签到领分、提交任务、兑换或查看本人排名。
- 校园身份至少包含姓名、学院、学号或工号、身份类型；手机号可选。
- 同一学号或工号不得被多个微信身份直接覆盖绑定，冲突需后台处理。
- 排行榜和大屏不得展示完整学号、手机号或其他敏感信息；个人榜使用脱敏姓名或昵称。
- 涉及宿舍或个人隐私的图片不得进入大屏轮播。

## 4. 角色与权限

| 角色代码 | 角色 | 可执行操作 | 明确禁止 |
| --- | --- | --- | --- |
| `GUEST` | 游客 | 查看公开活动和规则。 | 签到、任务、积分、兑换、本人排名。 |
| `PARTICIPANT` | 普通参与者 | 学生端全部参与功能。 | 审核、核销、修改规则。 |
| `STAFF` | 工作人员 | 本人授权点位的任务核验、循环登记、兑换核销。 | 修改任务规则、越权查看全部用户数据。 |
| `REVIEWER` | 审核员 | 审核分配范围内的任务提交，导出审核记录。 | 修改活动、奖品和人员权限。 |
| `REWARD_ADMIN` | 奖品管理员 | 维护奖品库存、核销、处理兑换异常。 | 修改任务与积分规则。 |
| `ACTIVITY_ADMIN` | 活动管理员 | 全部后台功能、人员授权、回滚和补录。 | 绕过操作日志直接改数据库。 |

所有后台写操作和工作人员操作必须写入操作日志，记录操作人、角色、动作、目标、前后状态、请求 ID、时间和必要备注。

### 5.5 大屏页面要求

- 刷新频率默认 60 秒，可配置为 30 至 300 秒。
- 大屏读取后端快照或缓存，不直接执行高成本实时聚合。
- 必须展示：参与人数、签到人数、任务完成数、积分发放量、学院榜、任务热度。
- 可展示：绿行累计步数、光盘通过数、减塑次数、循环数据、宿舍参与、答题正确率和估算贡献。
- 不展示完整姓名、学号、手机号、宿舍号和隐私图片。
- 接口异常时保留最后一次成功数据并显示“更新时间”，不得整屏空白。

### 6.2 任务规则结构

每个任务至少包含以下配置：

| 字段 | 含义 |
| --- | --- |
| `task_id`、`activity_id`、`zone_code` | 任务、活动和专区标识。 |
| `name`、`description`、`instructions` | 名称、说明和参与提示。 |
| `completion_type` | `AUTO`、`QR`、`PHOTO`、`STAFF_ENTRY`、`QUIZ`、`COMPOSITE`。 |
| `evidence_types` | 允许的证据类型列表。 |
| `review_mode` | `AUTO`、`MANUAL`、`SAMPLE`。 |
| `account_type` | 发放积分账户类型。 |
| `points_rule` | 固定值、阈值、计量档位等结构化规则。 |

| `daily_zone_points_cap` | 专区每日积分上限。 |

## 7. 状态机与关键流程

### 7.1 活动状态

`DRAFT -> PREHEAT -> ACTIVE -> ENDED`

- `DRAFT`：学生不可访问。
- `PREHEAT`：可查看预告、规则和奖品，不可签到、做任务或兑换。
- `ACTIVE`：按开关开放签到、任务、兑换和大屏。
- `ENDED`：业务只读，可查看成果、记录和证书；管理员可导出和处理经授权的纠错。

### 7.2 学生任务展示状态

`NOT_STARTED`、`IN_PROGRESS`、`PENDING_REVIEW`、`COMPLETED`、`REJECTED`、`LIMIT_REACHED`、`EXPIRED`

数据库中的提交记录不得复用展示状态替代审核状态。自动任务可直接生成 `APPROVED` 记录；照片类任务使用 `SUBMITTED -> APPROVED/REJECTED`。

### 7.3 兑换状态

`PENDING_REDEEM -> REDEEMED`  
`PENDING_REDEEM -> EXPIRED`  
`PENDING_REDEEM -> CANCELLED`

- 创建兑换时，在同一事务中校验并扣减指定账户余额、扣减可售库存、生成唯一兑换码。
- 核销只改变兑换状态，不重复扣库存和积分。
- 管理员取消待核销兑换时，在事务中恢复库存和余额并写反向流水。
- 已核销记录不得普通取消；确需处理时走管理员纠错流程并保留完整审计。

### 7.4 审核与发分流程

1. 接收提交并校验认证、活动、任务、时间、次数、上限和幂等键。
2. 保存任务记录、证据元数据和命中的规则版本。
3. 自动任务进入 `APPROVED`；人工任务进入 `SUBMITTED`。
4. 审核通过时调用积分引擎，积分引擎再次校验是否已发分。
5. 在同一事务中写任务状态、积分流水和操作日志。
6. 审核驳回时记录原因，不写正向积分流水。

### 7.5 积分引擎约束

- 输入只允许包含业务来源、用户、任务记录和规则版本，不接收前端积分值。
- 先校验业务来源唯一性，再计算积分，再校验专区上限。
- 达到上限时返回 `LIMIT_REACHED`，不写零分正向流水。
- 积分流水写入与任务完成必须处于同一事务，失败则整体回滚。
- 任何补发或扣减必须带原因、操作人和关联对象。

## 8. API 统一契约

- 基础路径：`/api/v1`

## 10. 安全、稳定性与可观测性

### 10.1 安全要求

- 所有权限由后端验证；隐藏前端按钮不能代替授权。
- 所有二维码校验签名、有效期、状态和业务上下文。
- 图片仅允许配置的类型和大小；上传后计算哈希并进行必要的内容安全处理。
- 导出文件按角色过滤字段，下载链接限时有效，并记录下载人和时间。
- 日志不得记录访问令牌、完整手机号、完整学号或图片原始隐私信息。
- 管理员回滚、补发、取消兑换等高风险操作要求二次确认和原因。

### 10.2 稳定性要求

- 核心写接口必须幂等，客户端超时重试不得产生重复任务、积分或兑换。
- 兑换、积分发放和管理员纠错使用数据库事务。
- 首页、排行榜和大屏使用缓存；缓存失效时可降级到最后成功快照。
- 图片上传失败可重试，不得因重复点击创建多条有效提交。
- 活动结束状态由后端统一拦截写操作。
- 保留人工补录入口，但必须由活动管理员执行并形成审计流水。

### 10.3 性能与监控验收

| 项目 | 完整验收目标 |
| --- | --- |
| 压测规模 | 模拟 100 至 500 人集中访问；包含签到、任务提交、查询和兑换。 |
| 普通查询 | 无外部上传时，P95 目标不高于 800 ms。 |
| 核心写接口 | 无图片传输时，P95 目标不高于 1500 ms。 |
| 大屏刷新 | 30 至 300 秒可配置，默认 60 秒。 |
| 监控 | 记录请求量、错误率、P95、数据库连接、缓存命中、待审核量、兑换库存和任务队列。 |
| 告警 | 登录失败激增、核心接口错误率、数据库或缓存异常、待审核积压、库存异常。 |

## 12. 测试与验收清单

### 12.1 完整上线门槛

- 微信登录、身份绑定和角色判断完整跑通。
- 同一学号绑定冲突不会覆盖原数据。
- 扫码签到只成功一次，积分流水只生成一次。
- 六大专区入口、任务状态、限制和提示正确。
- 自动、扫码、上传、人工审核、工作人员登记五类任务至少各有一个完整用例。
- 所有积分都有来源、规则版本、状态和时间。
- 兑换创建、并发库存扣减、核销、过期和管理员取消完整跑通。
- 工作人员只能操作授权点位和任务。
- 活动未开始、进行中、已结束时，页面和后端行为一致。
- 排行榜、大屏和导出使用同一统计口径。

### 12.2 必测异常用例

- 重复扫码、重复提交、重复审核、重复发分和重复核销。
- 客户端超时后重试同一请求。
- 二维码过期、停用、签名错误、点位错误和起终点顺序错误。
- 图片上传成功但业务提交失败，以及业务提交成功后客户端未收到响应。
- 审核通过后再次审核；管理员回滚后排行榜刷新。
- 两名用户同时兑换最后一件奖品。
- 规则修改后，历史记录仍使用旧规则版本。
- 活动结束瞬间仍有写请求到达。
- Redis 或大屏快照不可用时的降级行为。
- 无权限角色直接调用后台或工作人员接口。

### 12.3 数据核对

- 积分余额可由积分流水重算得到。
- 奖品剩余库存 = 初始/补充库存 - 有效兑换占用 + 取消恢复。
- 排行榜积分 = 有效正向任务积分 - 回滚，不受兑奖扣减影响。
- 大屏任务完成数仅统计有效完成状态。

## 13. 完整交付与完成定义

本文档中的学生端、工作人员端、管理后台、数据大屏、后端服务、数据库、接口、权限、监控和测试必须作为一个完整系统同时交付。开发过程可以由团队内部自行安排，但不得以开发顺序为由删减、推迟或单独验收任何本文档范围。

### 13.1 必须同时交付的成果

### 13.2 完成定义
